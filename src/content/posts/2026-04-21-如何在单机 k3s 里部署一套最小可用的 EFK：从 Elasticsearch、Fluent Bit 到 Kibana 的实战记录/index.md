这次我不打算再单独讲一篇“ELK 是什么”的概念文了。

原因很简单：

如果目标就是学 `k3s` / `Kubernetes`，那最值得先跑通的，不是传统 ELK 组合，而是更贴近容器场景的 **EFK**。

也就是：

- **Elasticsearch**
- **Fluent Bit**
- **Kibana**

这篇我就直接按“单机 k3s 怎么落一套最小可用 EFK”来写。

重点不是泛泛介绍概念，而是回答这几个实际问题：

- 为什么这次我不先选 Logstash
- 在 k3s 里，日志到底从哪来
- Elasticsearch、Fluent Bit、Kibana 三层分别放在哪里
- 单机实验环境里，最小方案应该怎么收敛

## 为什么这次我选 EFK，而不是 ELK

因为这次目标不是学传统日志平台，而是学 **Kubernetes 里的日志链路**。

如果在 K8s 里看问题，日志的起点一般不是“某个应用自己往某个路径里写一份文件，然后等你去收”，而是：

- 容器输出到 `stdout`
- 容器输出到 `stderr`
- 节点上有对应的容器日志文件
- 采集器再从节点统一收走

这时候，采集层最合适的往往不是 Logstash，而是：

- `Fluent Bit`
- 或者 `Fluentd`

我这里先选 `Fluent Bit`，原因就一个：

> **它更轻，更适合先在单机 k3s 里跑通一条日志链路。**

## 先把这套结构想清楚

如果只看这次最小实验，我实际想要的是这条链路：

```text
Pod 日志
  ↓
节点上的容器日志文件
  ↓
Fluent Bit（按节点采集）
  ↓
Elasticsearch（存储和索引）
  ↓
Kibana（查询和展示）
```

这条线里，每层职责都很清楚：

### 1. Fluent Bit
负责从节点读取容器日志，再把日志发给 Elasticsearch。

### 2. Elasticsearch
负责把日志存下来，并支持后续检索。

### 3. Kibana
负责把日志展示出来，让你能按字段和时间范围去查。

## 单机 k3s 下，我怎么收敛这次方案

如果一开始就想做很完整的日志平台，很容易把自己绕进去。

所以这次我给自己定的目标很克制：

- Elasticsearch 先单实例
- Kibana 先单实例
- Fluent Bit 先按 DaemonSet 跑在节点上
- 只要能把 Pod 日志送进去，并且在 Kibana 里查到，就算这一轮成功

也就是说，这次先不追求：

- 多节点 Elasticsearch
- 高可用
- 复杂日志解析
- 特别完整的索引生命周期策略

先把最小链路跑通。

## 这次部署时，我更看重的不是“会不会装”，而是部署方式别太偏

如果只是为了理解组件，你当然可以：

- `docker run elasticsearch`
- `docker run kibana`

但如果目标是学 `k3s`，那这种方式还是太偏离 Kubernetes 本身。

所以这次我更希望整件事至少保留下面这些味道：

- 后端服务按 K8s 工作负载部署
- 采集器按节点跑
- Kibana 通过 K8s Service 暴露
- 配置尽量通过 values 或 manifest 管，而不是靠临时手工改

这样后面继续扩展时，不至于完全推翻重来。

## Elasticsearch 这层，我怎么理解它在单机 k3s 里的位置

这层是整个日志系统的后端。

但放到单机实验里，它也最容易变成资源黑洞。

所以这次思路应该很明确：

- 不搞多节点
- 不搞太大的资源申请
- 先接受它只是一个单实例日志后端

对这一步来说，最重要的不是“参数调优多专业”，而是：

- 它能正常起来
- Kibana 能连它
- Fluent Bit 能把日志发进去

只要这三件事成立，这层就已经足够支撑学习。

## Kibana 这层，不是装完能打开就结束了

Kibana 的真正价值，不是“有个页面”，而是你最后能在里面按 Kubernetes 的视角查日志。

也就是说，到最后我想看到的不是一堆散乱文本，而是至少能围绕这些维度去看：

- namespace
- pod
- container
- 时间范围

这也是为什么前面 Fluent Bit 那层“带上 Kubernetes 元数据”很重要。

如果没有这些字段，Kibana 在 K8s 场景里的价值会打折很多。

## Fluent Bit 这层，才是这次最贴近 Kubernetes 的地方

如果只说“EFK 包含什么”，其实没有太大价值。

真正值得学的是：

> **为什么 Fluent Bit 很适合在 K8s 里作为采集层。**

因为它很适合做这件事：

- 每个节点跑一个实例
- 读取节点上的容器日志
- 加上 Kubernetes 元数据
- 再统一送去后端

也就是说，这层天然就适合用 **DaemonSet** 去理解。

这一点很关键。

因为它不是“某个应用的一个 sidecar”，而是：

> **整个节点层面的统一采集器。**

## 如果按实际部署顺序排，我会怎么做

这次如果真按动手顺序来排，我会这样分。

### 第一步：先把 Elasticsearch 起起来

先确保：

- Pod 正常
- Service 正常
- 存储至少是可用的

### 第二步：再把 Kibana 接上 Elasticsearch

先确保：

- Kibana 能启动
- Kibana 能连上 Elasticsearch
- 页面能访问

### 第三步：再上 Fluent Bit

这一步才真正让日志链路接通。

要验证的重点就变成：

- Fluent Bit 是否在节点上正常运行
- 它是否能读取容器日志
- Elasticsearch 里是否开始出现日志
- Kibana 里是否能查到数据

## 单机 k3s 的最小成功标准应该怎么定

如果不给自己定一个明确验收标准，很容易装了一堆东西，却不知道算不算完成。

所以这次我会把最小成功标准定成这样：

### 1. Elasticsearch 正常

至少后端服务是可用的。

### 2. Kibana 正常

至少页面能打开，并能正常连接 Elasticsearch。

### 3. Fluent Bit 正常

至少采集器在节点上是 Running。

### 4. Kibana 里能查到 Pod 日志

这是最关键的一条。

因为只有这一条成立，才说明：

- 采集通了
- 转发通了
- 存储通了
- 查询通了

## 为什么这条路线更适合现在这个阶段

因为你现在更需要的不是“掌握所有日志平台知识”，而是：

> **先把 Kubernetes 里的日志链路真正跑通一次。**

只要这一条通了，后面你再继续补：

- 更复杂的 parser
- 更细的字段处理
- 更成熟的索引策略
- 更完整的日志平台治理

都会顺很多。

但如果这条最基础的链路都没亲手跑通过，那前面背再多 ELK / EFK 概念，最后还是空的。

## 这次实战里，我会刻意避免什么

为了不把自己带偏，这次我会刻意避免下面这些做法：

### 1. 不一开始就把 Logstash 拉进来

因为现在最需要的是跑通 K8s 日志链路，不是上来就做重型日志处理。

### 2. 不一开始就追求“生产级日志平台”

单机 k3s 本来就只是实验环境，先把结构学明白更重要。

### 3. 不把重点放在大而全的功能上

先别急着追：

- 高可用
- 多节点
- 复杂索引生命周期
- 花哨 dashboard

先确保最小链路成立。

> **在单机 k3s 里，先别急着追完整日志平台，先把 EFK 这条最小日志链路跑通。**

这条链路一旦跑通，你对 Kubernetes 日志系统的理解就会立刻落地很多。

因为到那时候，你看到的就不再只是几个名词，而是一整条真的能工作的结构：

- Pod 产生日志
- Fluent Bit 采集日志
- Elasticsearch 存日志
- Kibana 查日志

这才是现在这个阶段最值得先学会的东西。
