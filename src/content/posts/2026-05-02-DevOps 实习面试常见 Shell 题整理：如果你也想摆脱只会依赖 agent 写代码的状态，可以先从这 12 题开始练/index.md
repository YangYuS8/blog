这篇我想写给两种人。

第一种，是已经开始往 DevOps 方向走，但写代码时越来越依赖 agent 的人。

第二种，是明明知道自己不是完全不行，却总会在最基础的语法、条件判断、循环、文本处理这些地方卡住的人。

我最近也在慢慢更明确地意识到一件事：

**如果我只是会“让 agent 帮我写”，但自己连最基础的 Shell 语法都不敢碰，那很多时候我并不是真的掌握了这项能力。**

尤其是想往 DevOps 走，这个问题会更明显。

因为 DevOps 不是只会点平台、会跑工具就够了。你迟早会碰到这些场景：

- 临时写一段部署脚本
- 看懂别人留下的运维脚本
- 改 CI/CD 里的 Shell 步骤
- 在服务器上快速排查日志和文件
- 把重复操作整理成可复用脚本
- 面试时被要求现场写几段简单脚本

这些事不一定复杂，但它们很少允许你一直停留在“我知道大概要怎么做，但我自己写不出来”的状态。

所以这篇我不打算直接写成“Shell 教程”，也不想上来就把标准答案全部摊开。

我更想整理一份：

> **偏 DevOps 实习面试语境下，比较常见、也比较值得自己动手练的 Shell 题目清单。**

这份清单更像是练习入口。

你可以先自己做，做不出来再去查资料、再去问 agent、再去对答案。重点不是第一遍就全做对，而是把“看到题完全不敢下手”慢慢变成“我知道该从哪一步开始拆”。

## 先说一句实话：你现在缺的，往往不是“高级技巧”，而是最基础的可执行感

很多人一提到 Shell 学习，脑子里先冒出来的是：

- awk
- sed
- xargs
- 正则
- 一行流式处理
- 很炫的命令链

这些当然重要，但如果你现在的状态是：

- `if` 还不熟
- `for` 和 `while` 容易写乱
- `$1`、`$#`、`$?` 不太清楚
- 文件判断和字符串判断总记混
- 一进面试就不知道该先写什么

那你最需要补的，通常不是“更高级”，而是“先把最常见的骨架写顺”。

也就是说，先让自己具备两种能力：

1. **能把题目翻译成脚本步骤**
2. **能把步骤写成最朴素但能跑的 Shell**

这比背很多花哨技巧更重要。

## DevOps 实习里为什么很爱考 Shell

原因其实很现实。

面试官不一定真的期待你写出多高级的脚本，但他们很想通过 Shell 看几件事：

- 你有没有 Linux 命令行的基本感觉
- 你能不能做最基本的自动化
- 你碰到重复操作时，会不会想着脚本化
- 你对文件、进程、日志、文本处理有没有基本认知
- 你是不是只会“背概念”，还是能真的动手

而且 Shell 题有个好处：

**很容易在短时间里看出候选人是不是自己写过东西。**

因为很多基础细节是装不出来的。

比如：

- 你知不知道脚本参数怎么取
- 你会不会做输入校验
- 你写循环时会不会漏引号
- 你知不知道 `grep` / `awk` / `cut` / `sort` / `uniq` 在什么场景下能快速解决问题
- 你有没有“先让它跑起来，再慢慢优化”的意识

所以如果你最近刚开始补 Shell，这方向其实是对的。

## 这篇怎么用

我下面会给 12 道题。

每道题我会写：

- 题目本身
- 它为什么常见
- 这题主要在考什么
- 你做的时候应该注意哪里

**我不会直接把完整答案全贴出来。**

因为如果一上来就看答案，这篇对你就更像“收藏夹”，而不是“训练器”。

我更建议你这样练：

1. 先限时自己写 10～20 分钟
2. 写不出来也别急着放弃，先把思路写出来
3. 再去查命令、补语法、问 agent
4. 最后自己重新敲一遍能跑通的版本
5. 隔几天再重做一遍，看看是不是还会卡

## 题目 1：判断一个文件是否存在，不存在就退出

### 题目

写一个 Shell 脚本，接收一个文件路径作为参数：

- 如果文件存在，输出 `file exists`
- 如果文件不存在，输出 `file not found` 并退出，退出码非 0

### 为什么常见

这是最基础、也最典型的 Shell 面试题之一。

在 DevOps 场景里，你经常要先确认：

- 配置文件在不在
- 日志文件在不在
- 某个脚本依赖的输入文件在不在

### 主要在考什么

- 位置参数，比如 `$1`
- `if` 条件判断
- 文件判断操作符
- `exit 1`

### 你做的时候要注意

- 先想清楚“没传参数”要不要处理
- 路径变量最好加引号
- 区分“普通文件存在”和“路径存在”不是一回事

## 题目 2：统计一个目录下有多少个 `.log` 文件

### 题目

写一个 Shell 脚本，接收目录路径作为参数，统计该目录下 `.log` 文件的数量，并输出结果。

### 为什么常见

因为日志文件、备份文件、构建产物的批量统计，是运维里非常常见的动作。

### 主要在考什么

- 参数接收
- 目录判断
- 通配符或 `find`
- 命令输出与计数

### 你做的时候要注意

- 想一想是只统计当前目录，还是递归统计子目录
- 如果目录不存在，应该怎么报错
- 空结果时脚本是否还能稳定运行

## 题目 3：循环输出 1 到 100 中的偶数

### 题目

写一个 Shell 脚本，输出 1 到 100 中所有偶数，每行一个。

### 为什么常见

这类题本身业务价值不大，但它非常适合看你对循环和算术判断熟不熟。

### 主要在考什么

- `for` 或 `while`
- 算术运算
- 条件判断

### 你做的时候要注意

- 不要一开始就追求写得最短
- 先写出一个最直白、最稳定的版本
- 想一想 Shell 里做数值比较和字符串比较有什么差别

## 题目 4：找出访问日志里出现次数最多的 IP

### 题目

给你一个 Nginx access log，找出访问次数最多的 IP，并输出它和出现次数。

### 为什么常见

这题非常贴近 DevOps 和运维实战。

你排查异常流量、刷接口、压测来源、恶意请求时，经常都要先看这个。

### 主要在考什么

- 文本处理思路
- `awk` / `cut` / `sort` / `uniq` / `head` 的组合使用
- 你能不能先观察日志格式，再决定从哪一列取字段

### 你做的时候要注意

- 不要先背命令链，先看日志格式
- 搞清楚 IP 在第几列
- 先拆成多步做，跑通后再考虑压成一行

## 题目 5：批量检查多个主机是否能 ping 通

### 题目

准备一个 `hosts.txt`，里面每行一个 IP 或域名。写一个脚本，逐行读取并检查每个主机是否能 ping 通，输出类似：

- `10.0.0.1 ok`
- `10.0.0.2 failed`

### 为什么常见

这类题很像最简化版的批量巡检。

### 主要在考什么

- `while read` 读取文件
- 调用外部命令
- 根据退出码判断成功失败
- 输出格式控制

### 你做的时候要注意

- `read` 循环怎么写更稳
- 判断命令是否成功时，要理解 `$?` 或直接跟 `if command; then`
- 想想空行要不要跳过

## 题目 6：监控磁盘使用率，超过阈值就报警

### 题目

写一个脚本，检查根分区使用率：

- 如果超过 80%，输出 `disk usage high`
- 否则输出 `disk usage normal`

### 为什么常见

这题很接近真实监控逻辑。

很多更复杂的监控脚本，最小核心也就是：

- 取值
- 比较
- 告警

### 主要在考什么

- 命令输出提取
- 数值比较
- Shell 里的字符串清洗

### 你做的时候要注意

- `df -h` 看起来方便，但机器读的时候不一定最省心
- 使用率里的 `%` 怎么去掉
- 别把字符串比较写成数字比较，或者反过来

## 题目 7：批量重命名文件

### 题目

把当前目录下所有 `.txt` 文件重命名为 `.bak` 文件。

例如：

- `a.txt` → `a.bak`
- `notes.txt` → `notes.bak`

### 为什么常见

这类题在脚本自动化里特别典型，因为它同时涉及循环、字符串处理和文件操作。

### 主要在考什么

- `for` 循环
- 参数展开
- `mv`
- 你是否知道 Shell 自带的字符串处理方式

### 你做的时候要注意

- 先想想如何只替换后缀
- 文件名里如果有空格，你的写法会不会炸
- 真实环境里最好先 dry run，但面试时至少要体现你知道这个风险

## 题目 8：统计某个进程是否存在，不存在就提示

### 题目

写一个脚本，检查某个进程（例如 `nginx`）是否在运行：

- 如果在运行，输出 `nginx is running`
- 如果没运行，输出 `nginx is not running`

### 为什么常见

服务健康检查是 DevOps 里特别常见的事情。

### 主要在考什么

- 进程查询命令
- 条件判断
- 命令退出码

### 你做的时候要注意

- 不要一上来就写得太复杂
- 想一想 `ps`、`pgrep`、`pidof` 之类工具在语义上有什么区别
- 注意误匹配问题

## 题目 9：从日志中筛出 ERROR 行并统计数量

### 题目

给定一个应用日志文件，筛出包含 `ERROR` 的行，并统计总共有多少条。

### 为什么常见

日志排查是 DevOps 的日常动作，这题几乎属于“必须会一点”。

### 主要在考什么

- `grep`
- 管道
- 计数
- 文本筛选的基本功

### 你做的时候要注意

- 大小写是否敏感
- 是只统计条数，还是同时输出内容
- 如果文件不存在，要不要先拦截

## 题目 10：读取一个配置文件，提取 key=value 中的值

### 题目

假设有这样一个文件：

```text
APP_ENV=prod
APP_PORT=8080
APP_DEBUG=false
```

请写脚本提取 `APP_PORT` 的值。

### 为什么常见

因为运维脚本、CI 变量处理、环境配置读取，经常都会碰到这种简单格式。

### 主要在考什么

- 文本过滤
- 分隔符处理
- `grep` / `cut` / `awk` 等工具的基础使用

### 你做的时候要注意

- 别假设文件永远完美无误
- 如果 key 重复出现怎么办
- 如果值里也带 `=`，你当前方案还稳不稳

## 题目 11：备份一个目录，并带上当天日期

### 题目

写一个脚本，把 `/data/app` 打包备份到 `/backup`，文件名里带上当天日期。

比如：

`app-2026-05-02.tar.gz`

### 为什么常见

这题很贴近真实运维操作：

- 打包
- 归档
- 日期变量
- 路径拼接

### 主要在考什么

- 日期命令
- 变量拼接
- `tar`
- 输出文件命名

### 你做的时候要注意

- 日期格式别手写，尽量让命令生成
- 源目录和目标目录要区分清楚
- 打包前要不要判断目录是否存在

## 题目 12：把脚本执行结果写入日志文件

### 题目

写一个脚本，执行一条命令（例如 `df -h`），并把结果追加写入某个日志文件，同时记录执行时间。

### 为什么常见

这是自动化脚本里特别常见的收尾动作。

很多脚本本体不复杂，但如果没有日志，后面排查会很痛苦。

### 主要在考什么

- 重定向
- 追加写入
- 日期时间命令
- 基本日志意识

### 你做的时候要注意

- 覆盖写和追加写的区别
- 标题行和正文如何分开
- 时间戳最好格式统一

## 如果你想把这 12 题练出效果，建议按这个顺序来

如果你现在基础还不稳，我更建议分三轮来做。

### 第一轮：先把骨架写顺

先做这些：

- 题 1：文件判断
- 题 3：循环与数字判断
- 题 5：读文件 + 判断退出码
- 题 8：进程检查
- 题 12：日志追加

这一轮目标不是优雅，而是：

**把最基础的 Shell 结构写顺。**

### 第二轮：开始做一点文本处理

再做这些：

- 题 2：统计文件
- 题 9：筛 ERROR
- 题 10：提取配置值
- 题 11：备份目录

这一轮你会开始频繁接触：

- 管道
- 文本提取
- 变量拼接
- 命令替换

### 第三轮：做更贴近实战的题

最后重点做这些：

- 题 4：日志里找最高频 IP
- 题 6：磁盘使用率判断
- 题 7：批量重命名

这几题更容易暴露出你对：

- 文本处理
- 数值判断
- 参数展开
- 命令组合

到底是不是已经有感觉了。

## 做这些题的时候，我最建议你强迫自己养成的 4 个习惯

### 1. 先写伪代码，再写 Shell

不要一看到题就直接敲命令。

你可以先用自然语言写三四步：

1. 取输入
2. 判断输入是否合法
3. 执行命令
4. 格式化输出

很多时候，卡住不是因为 Shell 太难，而是因为脑子里步骤本来就没拆开。

### 2. 每次都先考虑异常情况

比如：

- 没传参数怎么办
- 文件不存在怎么办
- 目录不存在怎么办
- 命令执行失败怎么办
- 空输入怎么办

面试里很喜欢看这个。

因为这能区分：

- 你只是“把 happy path 拼出来了”
- 还是你真的在按脚本思维做事

### 3. 变量尽量加引号

这是特别基础、但也特别容易被忽略的习惯。

很多脚本在你本机上看着没问题，只是因为你的输入比较“干净”。

一旦路径里有空格、特殊字符，很多没加引号的写法就会炸。

### 4. 不要一开始就追求一行流

很多人一学 Shell 就会被“一行命令很帅”吸引。

但真实学习路径通常更适合这样：

- 先分步骤写
- 能跑通
- 再慢慢压缩
- 再优化可读性

**先可运行，再可炫技。**

## 最后想说一句

如果你现在已经开始意识到：

> “我不能一直停留在完全依赖 agent，自己却连基础语法都不敢碰。”

那其实这已经是一个很好的开始了。

因为最难的通常不是某个语法点本身，而是你愿不愿意承认：

**自己确实需要把底层基本功补起来。**

Shell 很适合作为这个突破口。

它离 Linux 很近，离 DevOps 日常很近，反馈也足够快。

你今天多手写几道这种题，可能短时间内还不会让你变成“很会写代码的人”，但它会慢慢把你从：

- 看题就怕
- 看到脚本就想躲
- 一离开 agent 就不会动

带到另一种状态：

- 我知道该先判断什么
- 我知道要不要用循环
- 我知道怎么处理输入和输出
- 我就算写得慢一点，也能自己把第一版弄出来

这一步，对想走 DevOps 的人来说，很重要。
